# LTL Model Checker

This repository implements an LTL (Linear Temporal Logic) model checking algorithm utilizing an automata-theoretic approach. It successfully parses a Transition System (TS) and LTL formulas, builds the corresponding automata, constructs their product, and verifies specific properties by identifying persistent cycles.

---

## 1. Compilation and Usage

**Dependencies**:
- A compiler supporting **C++20**.
- **CMake** (>= 3.22)
- **Boost** (>= 1.70) (Required for `boost::dynamic_bitset`).
- **fmt** (Modern formatting library).
- **ANTLR4 C++ Runtime** (For LTL parsing).

**Compilation**:
Run the following commands in the root directory:
```bash
mkdir build && cd build
cmake ..
make
```

**Usage**:
Run the compiled executable with the TS file and the LTL benchmark file as arguments:
```bash
./build/LTL_checker <TS.txt> <benchmark.txt>
```

Testing example inside the project:
```bash
./build/LTL_checker benchmark/1.ts.in benchmark/1.ltl.in
```

The output will be a sequence of `1`s and `0`s printed line by line, indicating whether each LTL formula is satisfied (`1`) or not (`0`).

**Automated Testing Script**:
For convenience and automated grading, a Bash script (`test.sh`) is provided in the root directory. This script performs the complete lifecycle of compiling the code, running testing over benchmarks, asserting correctness by verifying outputs against expected targets (the `.ans` files), and subsequently cleaning up.
```bash
bash test.sh
```
This executes `./build/LTL_checker benchmark/$i.ts.in benchmark/$i.ltl.in` across the testing benchmark scope (targets 1 through 4) automatically verifying full alignment with the assignment criteria.

## 2. Code Structure & Data Structure

- **`grammar/`**: Contains the ANTLR4 definition (`LTL.g4`) and generated C++ Lexer/Parser used to parse LTL formulas into parse trees.
- **`include/ast/`** & **`src/ast/`**: Contains abstract syntax tree representations. The `ASTBuilder` transforms the ANTLR parse tree into our internal AST representation with nodes natively supporting formula evaluation logic (like `NextASTNode`, `UntilASTNode`, `AndASTNode`).
- **`include/ts.h`** & **`src/ts.cpp`**: Provides the structures to store States, Labels (Atomic Propositions), and Transitions for the Transition System (`TransitionSystem` template class).
- **`include/gnba.h`**: Defines the Generalized Non-deterministic Büchi Automaton (`GNBA`). Uses a specialized nested template structure parameterized recursively by state type (which might be dynamic bitsets or tagged tuples).
- **`include/ltl_to_gnba_builder.h`** & **`src/ltl_to_gnba_builder.cpp`**: Evaluates Elementary Sets using `boost::dynamic_bitset<>` representing variable assignments on subformulas and constructs the GNBA.
- **`include/product_ts_constructor.h`** & **`src/product_ts_constructor.cpp`**: Defines logic to compute the synchronous product of the original TS and a standard NBA.
- **`include/persistence_checker.h`** & **`src/persistence_checker.cpp`**: Contains the model checking traversal core implementing the nested DFS.

## 3. Implementation Details for the 5 Key Steps

### Step 1: Parser (TS & LTL Formula Reading)
- **Code Locations**: `src/ts.cpp` (`ReadTransitionSystemInput`), `src/ast/ast_builder.cpp` (`ReadLTLInputFile`).
- **Description**: The parsed Transition System ignores terminal states based on constraints and processes custom formatted inputs mapping states to sets of atomic propositions (`a`, `b`, `c`, etc.). For LTL, the input strings are directly consumed by our ANTLR4 lexer and parser. Standard equivalences are handled internally during the AST building (e.g., resolving `impl` to `not` & `or`/`and`). Parentheses are strictly mandated by grammar to eliminate ambiguities.

### Step 2: Equivalent Construction from LTL Formula to GNBA
- **Code Location**: `src/ltl_to_gnba_builder.cpp` (`LTLToGNBABuilder::Build`).
- **Description**: Follows the textbook methodology. We treat each unique sub-formula ID from the AST as an index in an Elementary Set (`dynamic_bitset`). The algorithm evaluates constraints bottom-up inside `ComputeElementarySets()`. Successors are constructed precisely based on logical rules:
  - For `Next` $\phi$, the truth assignment must temporally match in the destination elementary set.
  - For `Until` ($\phi \ U \ \psi$), consistency is guaranteed among current state evaluation and successor evaluation via masking logic.
- Acceptance sets are then populated for every `Until` subformula by capturing states where either the `Until` target ($\psi$) evaluates to true immediately, or the broader formula evaluates to false.

### Step 3: Equivalent Conversion from GNBA to NBA
- **Code Location**: `include/gnba.h` (`ToNBA() const`).
- **Description**: To eliminate multiple acceptance sets ($F_0, \dots, F_{k-1}$) generated by generalized Büchi features without modifying semantics, we employ **Round-Robin State Duplication**.
  - A new state is a tuple: $Q' = Q \times \{0, 1, \dots, k-1\}$.
  - Moving from state $(q, j)$ to $(q', j)$ normally occurs, except when $q$ is found within the $j$-th terminal set ($F_j$), which promotes the state onto $(q', (j+1) \bmod k)$.
  - The comprehensive final acceptance set becomes merely $F_0 \times \{0\}$.

### Step 4: Product Construction (TS & NBA)
- **Code Location**: `src/product_ts_constructor.cpp` (`ProductTSConstructor::Build`).
- **Description**: Maps the unrolled semantic behaviors combining actual execution paths from the TS and requirement-checking boundaries provided by the NBA (`Definition 4.62, Page 200`).
  - Nodes of the product system represent pairings $(s, q)$.
  - The construction logic explicitly iterates transitions $s \to t$ in the local system combined with valid transitions $q \to p$ in the NBA, explicitly synchronizing on the criteria $p \in \delta(q, L(t))$. 

### Step 5: Nested Depth-First Search for Persistence
- **Code Location**: `src/persistence_checker.cpp` (`PersistenceChecker::Check`).
- **Description**: Implements textbook **Algorithm 8 (Page 211)**. Checking if the `Product TS` upholds the `Persistence Property` relies directly on a double DFS:
  1. **Outer DFS**: Performs a standard reachability traversal across the product states. Once it fully visits an expanded node's immediate neighborhood (backtracking from the stack), it assesses if the current state sits in the newly constituted NBA acceptance set.
  2. **Inner DFS**: If a target condition triggers, the algorithm halts normal sweeping and pushes the inner routine (`CycleCheck`). The Inner DFS seeks any path cycling back cleanly to this exact starting node. If affirmative, a persistent refuting cycle is discovered—immediately violating the temporal guarantees, yielding `false`. If cleanly scanned without cycles, it ultimately yields `true`.
